# ScoreCaddie 화면 설계

총 14개 화면 (사용자 10 + 관리자 4). 대시보드가 사용자 화면의 허브 역할을 하며 4대 기능(골프장 / 스코어 등록 / 스코어 조회 / 마이페이지)으로 분기하는 구조.
관리자 화면(11~14번)은 `User.role = ADMIN`인 계정만 접근 가능한 별도 영역으로, 골프장의 루프(구간)·루프별 홀 Par(규정타수) 데이터 등록/관리(11~13번)와 회원 관리(14번)를 다룬다.
DB 스키마 근거: `prisma/schema.prisma` (User, GolfCourse, GolfCourseLoop, GolfCourseHole, Round, HoleScore + NextAuth 테이블)

---

## 1. 랜딩

- **목적**: 비로그인 사용자 진입점. 서비스 소개 + 로그인/회원가입 유도.
- **경로**: `/`
- **레이아웃**: 상단 히어로(서비스명 + 한 줄 소개) → 핵심 기능 3~4개 카드 요약 → 로그인/회원가입 CTA 버튼
- **주요 컴포넌트**: 히어로 섹션, 기능 카드, CTA 버튼 2개
- **데이터 의존성**: 없음 (정적)
- **상태**: 로그인 상태면 `/dashboard` 로 자동 리다이렉트

## 2. 로그인

- **목적**: 이메일/비밀번호 또는 소셜(구글·카카오) 로그인
- **경로**: `/login`
- **레이아웃**: 중앙 카드 — 이메일/비밀번호 입력 폼 + 구분선 + 소셜 로그인 버튼 2개 + 회원가입 링크
- **주요 컴포넌트**: 입력 폼(이메일, 비밀번호), 에러 메시지 영역, 소셜 로그인 버튼(Google, Kakao), "회원가입" 링크
- **데이터 의존성**: NextAuth `signIn()` (Credentials / Google / Kakao)
- **상태**: 로딩(제출 중 버튼 비활성화), 에러(잘못된 이메일/비밀번호)

## 3. 회원가입

- **목적**: 신규 계정 생성 (이메일/비밀번호 기반, 개방형 가입)
- **경로**: `/signup`
- **레이아웃**: 중앙 카드 — 이름, 이메일, 비밀번호, 비밀번호 확인 입력 폼 → **타인에게 정보 제공 동의(선택) 체크박스** → 가입하기 버튼
- **주요 컴포넌트**: 입력 폼 4개 필드, 타인에게 정보 제공 동의 체크박스(선택 항목, 미체크 상태가 기본값)와 안내 문구, 유효성 에러 메시지, 제출 버튼, 로그인 페이지 링크
- **데이터 의존성**: `POST /api/signup` → `User` 테이블 (bcrypt 해시 저장, `thirdPartyConsent`는 체크박스 값 그대로 저장 — 2026-07-16 신규)
- **상태**: 이메일 중복 에러, 비밀번호 규칙 미달 에러, 제출 중 로딩, 동의 체크박스는 미체크여도 가입 절차를 막지 않음(선택 동의이므로 필수 검증 대상 아님)

## 4. 대시보드 (허브)

- **목적**: 로그인 후 첫 화면. 4대 기능으로 분기 + 최근 활동 요약
- **경로**: `/dashboard` (보호 라우트)
- **레이아웃**: 상단 인사말(사용자명) → 4개 대형 메뉴 카드(골프장 / 스코어 등록 / 스코어 조회 / 마이페이지) → (`role=ADMIN`인 경우) 관리자 배너 1줄 → 하단 최근 라운드 3건 미리보기
- **주요 컴포넌트**: 메뉴 카드 4개(아이콘+제목+설명), 관리자 배너(조건부 노출, 클릭 시 11번 화면으로 이동), 최근 라운드 리스트(골프장명·날짜·총타수)
- **데이터 의존성**: `Round` (최근 3건, `userId` 기준 `playedAt desc`), `GolfCourse`, `HoleScore` (합산 타수 계산용), `User.role` (관리자 배너 노출 여부)
- **상태**: 최근 라운드 없을 때 "첫 라운드를 등록해보세요" 빈 상태, `role=ADMIN`이 아니면 관리자 배너 미노출
- **화면 이동**: "스코어 등록" 카드 클릭 시 **7-1(Step1, 코스 선택)** 화면 호출 (파라미터 없이 진입 → 골프장 미지정 상태에서 시작), `role=ADMIN`이면 관리자 배너 2개 노출 — **11번(골프장 Par 관리)**, **14번(회원 관리)**로 각각 이동

## 5. 골프장 목록

- **목적**: 공공데이터 연동 골프장 검색/탐색
- **경로**: `/courses`
- **레이아웃**: 상단 검색창(이름/지역) → 목록(카드 또는 테이블: 골프장명, 주소, 지도 아이콘) → 페이지네이션
- **주요 컴포넌트**: 검색 입력, 필터(지역/공공·민간), 골프장 카드 리스트, 페이지네이션
- **데이터 의존성**: `GolfCourse` (name, address, latitude/longitude, businessStatus)
- **상태**: 검색 결과 없음, 좌표 결측(`needsGeocoding=true`)인 경우 지도 아이콘 대신 "위치 확인 중" 표시
- **화면 이동**: 골프장 항목 클릭 시 **6번(골프장 상세)** 화면을 `?course=<골프장명>` 파라미터와 함께 호출(실제 구현 시 `golfCourseId`) — 6번 화면은 이 값으로 해당 골프장 정보를 렌더링

## 6. 골프장 상세

- **목적**: 특정 골프장 정보 확인 + 해당 골프장에서의 내 라운드 이력
- **경로**: `/courses/[id]`
- **레이아웃**: 상단 골프장명/주소/지도 → "이 골프장에서 스코어 등록" 버튼 → 이 골프장에서의 내 라운드 이력 리스트
- **주요 컴포넌트**: 지도(좌표 기반), 골프장 정보 카드, 라운드 이력 리스트(날짜·총타수)
- **데이터 의존성**: `GolfCourse` (단건), `Round` (해당 `golfCourseId` + `userId` 필터)
- **상태**: 좌표 없음 → 지도 대신 주소 텍스트만 표시
- **진입 파라미터**: 5번(골프장 목록)에서 `?course=<골프장명>`으로 진입 시 해당 골프장 정보로 렌더링(지도/주소/홀수/공공·민간, 좌표 결측 골프장은 지도 영역에 "좌표 확인 중" 표시), 파라미터 없이 진입하거나 매칭 실패 시 첫 번째 골프장으로 대체 표시
- **화면 이동**: "이 골프장에서 스코어 등록" 버튼 클릭 시 **7-1(Step1)** 화면을 현재 보고 있는 골프장의 `?course=<골프장명>` 파라미터와 함께 호출 — Step1의 골프장 select가 이 값으로 미리 선택된 상태로 진입(실제 구현 시 골프장명 대신 `golfCourseId`로 전달)

## 7. 스코어 등록 (2026-07-16 2-Step 구조로 재설계)

- **목적**: 골프장·코스·일자 선택(Step1) 후 홀별 상세 스코어 입력(Step2)
- **경로**: `/rounds/new` (내부적으로 Step1/Step2 두 화면, 선택값은 쿼리 파라미터로 전달)
- **목업**: `doc/mockups/07-round-new-step1.html`, `doc/mockups/07-round-new-step2.html` (사용자 제공 참조 앱 "Even Par" 스타일 반영. 구버전 단일화면 목업 `07-round-new.html`은 비교용으로 보존)
- **진입 경로**: (1) 4번 대시보드 "스코어 등록" 카드 → Step1 파라미터 없이 진입, (2) 6번 골프장 상세 "이 골프장에서 스코어 등록" 버튼 → Step1에 `?course=` 파라미터로 골프장 미리 선택, (3) 9번 라운드 상세 "수정" 버튼 → **Step2로 직접** 기존 라운드 데이터를 파라미터로 채워 진입(아래 7-2 참고)

### 7-1. Step1 — 코스 선택

- **레이아웃**: 골프장 선택 → 홀 수(9H/18H) 토글 → 전반(1~9홀) 코스 선택 + (18H인 경우만) 후반(10~18홀) 코스 선택 → 라운드 일자 → 라운드 당시 날씨 자동 표시 → "스코어 카드" 버튼
- **주요 컴포넌트**: 골프장 콤보박스, 9H/18H 세그먼트 버튼, 전반/후반 코스 선택(9H면 후반 숨김), 날짜 피커, 날씨 카드(자동)
- **데이터 의존성**: `GolfCourse`(선택용), `GolfCourseLoop`(선택된 골프장의 루프 목록 — 전반/후반 select 옵션으로 표시, `sortOrder` 순), 날씨 API(좌표 기반 스냅샷 미리보기)
- **상태**: 9H 선택 시 후반 코스 선택란 숨김, 선택한 골프장에 루프가 1개도 등록되어 있지 않으면 코스 선택란 대신 "코스 정보 없음(관리자 등록 필요)" 안내 후 루프 선택 없이 진행 가능(→ `Round.frontLoopId/backLoopId`는 null로 저장)
- **진입 파라미터**: `?course=<골프장명>` — 6번 화면에서 넘어온 경우 골프장 select를 미리 선택(목업에서는 골프장명 문자열, 실제 구현 시 `golfCourseId`)
- **다음 화면 호출**: "스코어 카드" 버튼 클릭 시 **7-2(Step2)**를 `?holes=&course=&date=&frontLoop=&backLoop=` 파라미터와 함께 호출(`frontLoop`/`backLoop`는 선택된 `GolfCourseLoop.id`)

### 7-2. Step2 — 스코어카드 입력

- **레이아웃**: 전반/후반 스코어카드 표(홀 번호·Par·타수, 홀 탭으로 이동) → PAR 3/4/5/6 버튼(선택된 홀의 Par 즉시 변경) → 4분할 입력 패널 → 홀메모 → "전반/후반 N번홀 입력" 저장 버튼 → 하단 이전/다음/초기화 네비게이션
- **4분할 입력 패널**:
  1. **티샷 결과**: 페어웨이 / 페어실패 / 페널티 / OB 중 선택. **페널티 선택 시 페널티 타수 +1·온그린 타수 +1, OB 선택 시 OB 타수 +2·온그린 타수 +2가 자동 반영**되며(다른 결과로 바꾸면 이전 자동가산은 되돌리고 새로 적용), 이후 수동 조정 가능
  2. **페널티/OB**: 페널티 타수·OB 타수 스테퍼(◀▶), 그린주변벙커 체크박스
  3. **온그린 핀거리**: 0~20m는 1m 단위 셀렉트(21개 옵션), 20m 이상은 별도 버튼 — 둘 중 하나만 선택
  4. **스코어**: 온그린 타수·퍼트 타수 스테퍼, 오버파(+/E/-) 자동 계산 표시
- **주요 컴포넌트**: 스코어카드 표 2개(전반/후반, 9H면 후반 숨김), Par 선택 버튼, 4분할 입력 패널(위), 홀메모 입력(100자 제한), 홀 저장 버튼(누르면 다음 홀로 자동 이동), "라운드 상세" 링크(9번으로 이동, 아래 화면 이동 참고), 이전/다음/초기화 버튼, 수정 모드 진입 시 상단 "기존 라운드 수정 중" 안내 배지
- **데이터 의존성**: `Round` 생성(`holesPlayed`, `frontLoopId`, `backLoopId` 포함 — Step1에서 선택한 `GolfCourseLoop` 참조), 홀별 초기 Par는 선택된 루프의 `GolfCourseHole.par`를 기본값으로 불러오되 화면에서 PAR 버튼으로 재조정 가능, `HoleScore` 홀 수만큼 생성(`strokes`, `par`, `teeShotResult`, `penaltyStrokes`, `obStrokes`, `bunkerUsed`, `pinDistanceType`, `pinDistanceMeters`, `onGreenStrokes`, `puttStrokes`, `memo`), 날씨 API(좌표 기반 스냅샷 → `Round.weatherSnapshot`)
- **상태**: 저장 중 로딩, 필수 홀 미입력 시 유효성 에러, 온그린 핀거리 미선택 등 선택 항목은 저장 허용(필수 아님), 루프 미선택 라운드는 Par 기본값 없이 PAR 버튼으로 직접 지정
- **진입 파라미터**:
  - Step1 경유(신규 등록): `holes`, `course`, `date`, `frontLoop`, `backLoop`
  - **9번 라운드 상세 "수정" 버튼 경유(기존 라운드 수정)**: 위 파라미터에 더해 `edit=1`, `scores=<홀별 타수 콤마 구분>`(예: `4,5,4,3,...`) — 스코어카드에 기존 타수가 이미 입력·저장된 상태로 채워짐. 과거 라운드는 `teeShotResult`/`pinDistanceType` 등 상세 필드가 없을 수 있어(해당 컬럼이 nullable인 이유) `strokes`만 복원하고 온그린/퍼트는 화면에서 임시로 역산해 보여주며, 티샷결과·핀거리 등은 값 없이 시작(사용자가 필요 시 채워 넣음)
- **화면 이동**: "라운드 상세" 링크(기존 "전체홀" 버튼을 2026-07-16 변경) 클릭 시 **9번(라운드 상세)**를 `course`, `date`, `score`(지금까지 저장한 홀들의 타수 합계), `weather`, `ownerId=me`, `owner` 파라미터와 함께 호출 — 홀을 저장하거나 초기화할 때마다 `score`가 최신 합계로 갱신됨. 실제 구현 시에는 저장되지 않은 라운드도 임시 조회가 가능해야 하므로 서버에 임시저장(draft) 또는 세션 상태 처리 방식을 별도로 설계해야 함(현재 목업은 클라이언트 상태만으로 시연)

## 8. 스코어 조회 (목록, 2026-07-16 조회조건 개편)

- **목적**: 라운드 기록을 기간/골프장/사용자 범위로 조회 — 기본은 내 기록만, 필요 시 정보제공에 동의한 다른 회원의 기록도 함께 조회
- **경로**: `/rounds`

## 9. 라운드 상세 (2026-07-16 홀별 스코어 표시 개편 — 매트릭스형으로 확정)

- **목적**: 특정 라운드의 홀별 타수/티샷결과/GIR/스크램블링/그린주변벙커 세이브 상세 확인 및 수정

- **진입 파라미터**: 8번(스코어 조회)에서 `course`, `date`, `score`, `weather`, `ownerId`, `owner`를 전달받아 상단 요약 카드를 구성. 파라미터가 없으면 데모용 기본값(레이크사이드 CC · 2026-07-12 · 88타) 표시

- **주요 컴포넌트**: 요약 카드, 범례 바, 홀별 스코어 매트릭스(전반/후반 카드 2개), 홀 메모 표시 패널, 수정/삭제 액션 또는 읽기 전용 안내
- **홀별 스코어 매트릭스 — 최종 확정안(2026-07-16, 검토했던 탭-상세형/세로 리스트형 대안은 제외)**:
  - 18홀을 전반/후반 2개의 표로 한 번에 보여주는 기존 스코어카드 형태를 유지하되 다음과 같이 가독성을 개선:
    - "특이사항" 행에 티샷결과 2글자 약어 칩(FW/MS/PN/OB, 색상 구분) + GIR 성공/실패 점 + 스크램블링(S)·샌드세이브(✓/✗) 보조 표시를 정리하고, 범례를 표 하단이 아닌 화면 상단에 항상 고정 노출
    - 홀/파/스코어 3개 행을 배경색·굵기로 구분(홀 행은 옅은 구분선, 파 행은 연회색 배경, 스코어 행은 진한 배경+굵은 글씨)해 행 종류를 한눈에 구별
    - 스코어 숫자에 골프 스코어카드 관례를 적용 — 버디는 동그라미, 이글 이하는 이중 동그라미, 보기는 네모, 더블보기 이상은 이중 네모로 숫자를 감싸 언더파/오버파를 시각적으로 표시(파는 표시 없음)
    - "기타" 행을 실제 용도에 맞게 "메모"로 이름 변경, 메모가 있는 홀의 📝 아이콘을 탭하면 표 아래 패널에 해당 홀의 메모 내용이 표시됨(다른 아이콘 클릭 시 전환)
    - 전반/후반 라벨에 골프장명을 함께 표시(예: "전반 · 레이크사이드 CC")
    - 전반 표와 후반 표를 각각 테두리 카드로 감싸 시각적으로 분리하고, 전반은 primary(초록)·후반은 accent(골드) 컬러바로 구분
    - 2번 홀(샌드세이브 실패)과 12번 홀(샌드세이브 성공)을 대조 예시로 구성해 성공/실패 표시가 실제로 어떻게 보이는지 한 화면에서 비교 가능
  - 공통 로직: GIR = 온그린 타수 ≤ (파-2), 스크램블링 = GIR 실패 홀에서 최종 타수 ≤ 파(성공 시에만 표시), 샌드세이브 = 그린주변벙커 사용 홀에서 최종 타수 ≤ 파 여부(성공/실패)로 산정 — 모두 저장된 `HoleScore` 필드(`onGreenStrokes`, `strokes`, `par`, `bunkerUsed`)로부터 화면에서 계산하는 파생값이며 DB에 별도 컬럼으로 저장하지 않음
- **데이터 의존성**: `Round` (단건, 소유자 검증), `HoleScore` (18건 — `strokes`/`par`/`teeShotResult`/`onGreenStrokes`/`puttStrokes`/`bunkerUsed`/`pinDistanceType`/`pinDistanceMeters`/`memo`), `GolfCourse`, `User`(열람자 소유 여부 판단)
- **상태**: 본인 라운드는 항상 열람/수정 가능. 본인 라운드가 아닌 경우, **8번의 "전체 회원" 조회에서 진입 + 소유자가 `thirdPartyConsent=true`인 경우에 한해 읽기 전용으로 열람 허용**(수정/삭제 버튼 숨김, "다른 회원의 라운드는 읽기 전용으로 열람할 수 있어요" 안내문 노출) — 목업(`09-round-detail.html`)에 실제 구현 완료(2026-07-16)
- **화면 이동**: "수정" 버튼 클릭 시(본인 라운드만 노출) **7-2(Step2)**를 `edit=1&course=&date=&holes=&scores=` 파라미터와 함께 직접 호출(Step1을 거치지 않고 바로 스코어카드로 진입 — 골프장/일자/홀수는 이미 확정된 값이므로 재선택 불필요, 실제 구현 시 `scores` 대신 `roundId` 하나만 전달해 서버에서 `HoleScore` 전체를 조회하는 방식으로 대체)

## 14. 관리자 - 회원 관리 (신규, 2026-07-16)

- **목적**: 전체 회원을 이름/가입일자로 조회하고, 어드민 권한을 부여/해제
- **경로**: `/admin/users` (관리자 전용 보호 라우트)
- **레이아웃**: 조회 조건(이름 검색 + 가입일자 범위) → 검색/초기화 버튼 → 전체 회원 수 → 회원 그리드(체크박스·이름·이메일·가입일·어드민) → 1건 이상 선택 시 하단에 일괄 액션 바 노출
- **주요 컴포넌트**: 이름 검색 입력, 가입일자 범위(시작~종료 날짜 피커), 회원 그리드 테이블(행별 선택 체크박스, 헤더의 전체선택 체크박스는 부분 선택 시 indeterminate 상태), "어드민" 컬럼(`role` 값을 텍스트 대신 체크박스로 표시 — `ADMIN`이면 체크, 읽기 전용이며 실제 변경은 하단 일괄 액션으로만 수행), 일괄 액션 바("어드민 권한 부여" / "어드민 권한 해제" 버튼, 선택 인원 수 표시)
- **데이터 의존성**: `User`(`name`, `email`, `createdAt`, `role`) 목록 조회, 일괄 액션은 `role` 필드 업데이트(`PATCH /api/admin/users/role` 등으로 다건 처리)
- **상태**: 검색 결과 없음 빈 상태, `name`이 `null`인 계정(소셜 로그인 등)은 "이름 없음"으로 표시, `role !== ADMIN` 접근 시 차단 후 대시보드로 리다이렉트, 본인 계정의 어드민 권한을 스스로 해제하는 경우에 대한 처리(확인 경고 또는 차단)는 구현 시 정책 결정 필요
- **화면 이동**: 4번 대시보드의 "회원 관리" 관리자 배너에서 진입

## 공통 컴포넌트 (여러 화면에서 재사용)

- **NavBar**: 로그인 상태에 따라 다르게 표시 (비로그인: 로그인/회원가입 버튼, 로그인: 대시보드/마이페이지/로그아웃, `role=ADMIN`이면 "관리자" 메뉴 추가 노출)
- **GolfCourseCard**: 골프장 카드 (5, 6번 화면)
- **RoundListItem**: 라운드 리스트 항목 (4, 8번 화면)
- **WeatherBadge**: 날씨 스냅샷 표시 (7, 8, 9번 화면)
- **EmptyState**: 데이터 없음 공통 안내 컴포넌트
- **AdminGuard**: `role !== ADMIN`이면 관리자 라우트 접근을 차단하는 서버 사이드 가드 (11~14번 화면 공통)
